# 78 — 桌宠预览动作碎块根因：双轨守卫误诊修复

**日期**: 2026-07-03
**主题**: 动作列表预览部分动作显示图片碎块；单轨重构根治

## 现象

普通交互（点击 `touch`、随机 `dance`/`sleep`）显示正常，但在设置页「动作列表」逐一**预览**时，部分动作只显示散落的图片碎块，无法组装出完整模型。

## 根因（脚本验证 chaijun.skel + pixi-spine 3.8 运行时）

用真实骨骼数据 + 真实 `computeWorldVertices` 做头无仿真，得到与 doc 76/77 相反的结论：

1. **动作本身没有问题**：全部 20 个动画在「单轨 + mix」下播放，附件包围盒均保持紧凑（约 260–360px），与 setup 姿态（261×335px）同量级。
2. **setup / idle 姿态并不散开**：doc 76 的核心判断「柴郡 setup 本身就是散开的」不成立，因此建立其上的双轨守卫是误诊。
3. **守卫才是元凶**：doc 76/77 的方案是 track 0 循环 idle、track 1 播动作，再用 `patchSpineApply/Update` 在每帧把 68 根骨骼**强制拉回 idle 快照**。这会锁死骨骼，却放任动作轨的 **deform（网格顶点变形）/ IK** 继续生效 → 网格顶点相对被锁死的骨骼飞散 = 碎块。
4. **症状分布吻合**：常用的 `touch`/`dance`/`sleep` 恰好无 deform，所以看着正常；而预览可触发全部 20 个，其中 `attack`/`attack_left`/`motou`/`victory`/`skill`/`move`/`move_left`/`stand`/`stand2` **带 deform**，一预览就碎。

## 修复（`src/pet/spinePet.ts` 单轨重构）

- **删除**整套守卫机制：`preserveAssembledPoseDuringAction`、`patchSpineApply`、`patchSpineUpdate`、`snapshotAssembledPose`、`restoreAssembledPoseGuard`、`assembledBones`/`assembledSlotAttachments*` 三个 Map、`clearPosePreserve`、`clearActionTrack`、`ensureIdleTrackLooping`、双轨 `ACTION_TRACK`。
- **单轨模型**：待机与一次性动作都播在 track 0。`playOneShot` 从当前姿态 `setAnimation(action, mix=0.15)`，随即 `addAnimation(idle, loop, 0)` 排队，动作播完自动 mix 回待机循环。
- `isActionPlaying()` 改为读 `actionPlaying` 布尔标志（由动作起止维护），随机调度器逻辑不变。
- boot 动画改走 `playOneShot`，统一回归路径；移除不再使用的 `IDLE_SETTLE_SEC`。
- 公共 API 面（`start`/`previewPlay`/`playAnimation`/`playTap`/`handleClick`/`configureAnimations`/`setPowerMode`/`setRenderPaused`/`resizeCanvas`/`dispose` 等）保持不变，`main.ts` 无需改动。

## 验证

- 头无仿真：全部 20 个动画在单轨生命周期（idle→动作→回 idle）内逐帧包围盒稳定，无飞散。
- `tsc --noEmit`：`spinePet.ts` 零错误（其余报错为改动前既存、与桌宠无关）。
- 待人工验证：重启 dev / 「立即更新」后，设置页逐一预览 `attack`/`motou`/`victory`/`stand` 等带 deform 动作 → 不应碎块。

## 局限与后续

- 单轨方案下动作会完整驱动骨骼（含 deform），表现比守卫方案（锁死骨骼）更还原动画本意。
- doc 76/77 的双轨守卫方案就此废弃；如后续换用其他 Spine 资源，直接沿用单轨模型即可。
- 既存无关问题（未在本次修复范围）：`PetActionSettings.tsx` 台词导入反馈误用 `message` 字段（应为 `title`/`detail`）、`PetPanel.tsx` 若干未使用变量、`main.ts` `root` 可能为 null 的类型告警。
