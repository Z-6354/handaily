## 问答二

### 问题

导入的文本是**参考信息**（非结构化），需要 AI 处理后再写入人设，不是原样作为 Skill。

### 答复

已改回复用人设工坊流水线，与 `character_preprocess` → `character_generate_skill` → `apply_to_persona` 一致：

1. **参考文本** → `persona-preprocess.md`（更新时若已有结构化资料则用 `persona-text-merge.md` 合并）
2. **思考模型** → 结构化 JSON（`CharacterProfileData`）
3. **`persona-skill-generate.md`** → Skill Markdown
4. 写入 `{id}.md` / `{id}.json` 并更新 manifest

**后端**：`persona_import_text` / `persona_import` 改为 async，内部 `persona_process_reference`；复用 `persona_builder` 的 `build_preprocess_prompt_from_text`、`build_merge_prompt_from_profile`、`save_processed_persona`。

**前端**：`PersonaTextImportForm` 按钮改为「AI 处理并创建/更新」，提示需配置思考模型。

**注意**：未配置思考模型时会提示「请先在设置中配置思考模型（用于解析参考文本并生成 Skill）」。

## 问答三

### 问题

导入人设失败：`无法解析 AI 返回的 JSON: EOF while parsing a value at line 1 column 0`

### 答复

根因：部分思考模型（DeepSeek Reasoner、Ollama thinking 等）把正文放在 `reasoning_content` / `thinking`，`content` 为空；原逻辑只读 `content`，得到空串后 JSON 解析报 EOF。

修复：
- 新增 `ai/response.rs` 统一提取 `content` / 数组 parts / `reasoning_content` / `thinking`
- 思考模型 `max_tokens` 提升至 8192（Ollama `num_predict` 同步）
- 参考文本超 24000 字自动截断
- 空响应与无 JSON 时给出更明确错误提示

## 问答四

### 问题

导入人设失败：`API 请求失败: error sending request for url (https://ark.cn-beijing.volces.com/api/plan/v3/chat/completions)`

### 答复

该错误发生在**收到 HTTP 响应之前**（网络/超时层），不是 401 鉴权问题。火山 Agent Plan 网关地址本身正确。

常见原因：
1. **请求超时**：人设导入需两次 AI 调用（解析 + 生成 Skill），参考文本较长时可能超过原 60 秒限制
2. **网络波动 / 代理**：偶发连接中断

修复：
- 思考模型请求超时提升至 **300 秒**，连接超时 30 秒
- 长任务失败时自动重试一次
- 错误信息区分超时、连接失败与其它请求错误
- 前端 loading 提示「约 1～3 分钟」

若仍失败：在设置中先「测试」火山 Agent Plan 供应商；思考模型建议选用对话类模型（如 `deepseek-v3`），`ark-code-latest` 偏代码生成，不适合结构化 JSON 任务。

### 问题

2. 报错：`无法解析 AI 返回的 JSON: EOF while parsing a value at line 132 column 19`

- 新增 `close_json_fragment` 尝试修复截断 JSON
- 解析失败时自动压缩重试一次
- 错误提示改为「JSON 不完整」并给出缩短文本建议
