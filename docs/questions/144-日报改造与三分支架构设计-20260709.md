# 144 - 日报改造与三分支架构设计 - 20260709

> 顶层设计文档。落在 `main` 分支,三个分支共享同一份设计基线。
> 目标:①重构日报/行为检测为「省资源 + 功能完整」;②派生「端云分离」分支;③派生「纯 Live2D」分支。

---

## 一、现状与问题

### 1.1 当前数据流(单端一体)

```
poller(2s) → writer.flush_segment(每个 segment)
  → analysis.enqueue_segment            [每段都进队列]
      → text::analyze  (规则,便宜)
      → 若 confidence < 0.45 → 截图 + vision AI   [贵]
  → period_scheduler(30s tick)
      → 触发点:启动+5m / 整点 / 长会话15m → AI 时段总结
  → report::gather/compose → (可选 AI 成稿) → wechat 推送
```

### 1.2 资源消耗痛点

- **逐段分析**:每次前台切换/内容变化都 `enqueue_segment`,即使多数段落语义重复(反复切 VSCode/浏览器)。文本分析便宜,但队列压力与 DB 写入(`activity_insights` 每段一行)偏高。
- **截图触发过宽**:`confidence < 0.45` 就截图,泛标题(浏览器、资源管理器)极易落到低置信度 → 频繁触发 vision AI(最贵项),仅靠 `screenshot_min_interval_secs=120` 与 CPU 阈值兜底,缺少「每日预算」与「场景冷却」。
- **AI 时段总结**:整点 + 长会话 + 启动都各自触发,重叠时段可能重复调用 AI。
- **无统一「行为会话」概念**:日报本质想回答"今天在学习/写代码/做别的",但底层是碎片 segment,AI 每次拿到的是零散片段,既贵又难汇总。

### 1.3 初衷复述

应用应像助手一样:**智能识别用户在做什么(学习/写代码/其他)**,并**定期汇报**。核心矛盾是"检测粒度 vs. 资源"。

## 二、日报改造方案(main 分支)

核心思想:**分层检测 + 事件驱动 + 预算约束**。把"识别在做什么"下沉到便宜的规则层,把"贵的 AI/截图"收敛为受预算控制的稀疏事件。

### 2.1 三层检测金字塔

| 层级 | 频率/触发 | 成本 | 职责 |
|------|-----------|------|------|
| L1 规则分类 | 每段(已有 `text::analyze`) | 极低 | 归类 coding/learning/other,给置信度 |
| L2 会话聚合 | 内存滚动,无 AI | 低 | 把连续同类 segment 合并成「行为会话」,只在会话**结束/切换**时落一条 insight |
| L3 AI 增强 | 事件驱动 + 预算 | 高 | 仅对"长会话 / 低置信 / 用户请求汇报"调用;截图与 vision 受日预算约束 |

关键改动:**insight 不再每段一行**,而是按「行为会话」落盘;L3 只对显著会话补充语义。

### 2.2 事件驱动替代轮询

- 保留 `period_scheduler` 的整点/启动触发,但**去重**:同一 [start,end] 区间只跑一次 AI(以 DB `periods` 是否已存在为准)。
- 长会话触发改为「会话结束时」单点触发,替代 30s tick 反复评估。
- 新增「用户主动请求汇报」路径(桌宠点击 / 微信指令 / IPC),按需生成,不占常驻预算。

### 2.3 预算约束(新增)

在 `AnalysisSettings` 增加(带默认值,可在设置页调):

- `screenshot_daily_budget`(默认 40):每日截图+vision 上限,超出仅走文本。
- `ai_period_daily_budget`(默认 24):每日 AI 时段总结上限。
- `session_min_ms_for_ai`(默认 10min):低于此时长的会话不触发 AI。
- 计数落 `daily_metrics` 或新表,每日 0 点重置。

预算耗尽后系统**自动降级**为纯规则模式,保证"功能完整"(仍能出日报),只是少了 AI 润色/截图细节。

### 2.4 兼容与迁移

- 不改 DB schema 的破坏性字段;新增预算计数以 `settings`/`daily_metrics` 承载。
- `report::gather` 逻辑不变(仍读 periods + timeline),因此前端与微信推送无需改。
- 改造集中在 `analysis/coordinator.rs`(会话聚合 + 预算门)、`analysis/period_scheduler.rs`(去重 + 事件化)、`analysis/mod.rs`(设置项)。

## 三、分支二:端云分离(feat/client-server-split)

### 3.1 目标

- **用户本地端**:只做信息收集(tracker + SQLite 缓存 + 上传),极简、常驻轻量。
- **服务器端(2 核 2G)**:挂载 Agent(analysis + AI + report + wechat),通过 API 通信。
- 用户日常聊天**无需开电脑**:微信侧由服务器常驻响应。

### 3.2 架构

```
[本地采集端 (Tauri 瘦身)]
  tracker → SQLite(本地缓冲) → 批量上报 HTTPS → [服务器 Agent]
                                                    ├─ 接收/入库 segment
                                                    ├─ analysis + AI 时段总结
                                                    ├─ report 生成
                                                    └─ wechat 常驻推送/应答
```

### 3.3 复用策略(关键:不重写业务逻辑)

现有 Rust 核心已模块化(`analysis` / `report` / `ai` / `wechat` / `db`),且已有 `agent_http.rs` HTTP 骨架。方案:

1. 把 `analysis/report/ai/wechat/db` 编译为**可在 Linux headless 运行**的二进制(新增 `[[bin]] xiaohan-agent`),去掉 Win32(`tracker/win32` 等)与 Tauri 窗口依赖(用 `cfg(feature)` 或 workspace 拆分)。
2. 本地端保留 `tracker` + 一个**上报客户端**,把 segment/metrics 批量 POST 到服务器。
3. 服务器扩展 `agent_http` 的路由:`POST /api/ingest/segments`、`GET /api/report/today`、微信 webhook。
4. 认证:设备 token(HMAC),TLS 由反代(caddy/nginx)承担,Agent 只听 127.0.0.1。

### 3.4 2C2G 资源约束

- Agent 用 tokio current-thread + SQLite(已有),内存目标 <300MB。
- AI 调用走远程供应商 API(不本地推理),CPU 压力主要在文本处理,可控。
- 截图/vision 在此分支**默认关闭**(本地端不上传截图,保护隐私 + 省带宽);语义靠标题文本 + AI。

### 3.5 版本管理

- 该分支 = main(改造后) + 采集/服务拆分。定期 `git merge main` 吸收日报改造成果。

## 四、分支三:纯 Live2D(feat/live2d-only)

### 4.1 目标

去掉日报、去掉 AI,只保留:模型挂载 + Live2D 加载 + 台词(用 wiki 爬取的)。做一个纯桌宠。

### 4.2 保留 / 移除

| 保留 | 移除 |
|------|------|
| `pet/`(桌宠窗口、模型加载、动画) | `analysis/`、`report/`、`ai/` |
| `pet/wiki_scrape.rs`(wiki 台词爬取) | `tracker/` 行为采集(可保留 idle 供动画,但不落库统计) |
| `pet/lines_import.rs`、`pet/models.rs` | `wechat/`、`period_scheduler`、`insights`/`periods`/`reports` DB |
| `character/`、`live2d`、`assets` | 报表 UI 页面、AI 供应商设置页 |
| Live2D/Spine 前端渲染 | AI 台词建议(`pet_ai_suggest_lines` 等) |

### 4.3 台词来源

- 仅 `pet_wiki_import_lines`(wiki 爬取)+ 手动导入;去掉 `pet_ai_suggest_lines`/`pet_ai_import_lines`。
- 台词随机播放由本地逻辑驱动,无需 AI。

### 4.4 瘦身收益

- 去掉 reqwest 对 AI 的依赖路径、去掉截图/vision、去掉微信,Cargo 依赖与二进制体积显著下降。
- 前端只保留桌宠 + 人物/皮肤/模型管理页。

### 4.5 版本管理

- 该分支基本不回并 main 的日报改造(方向相反);仅在 `pet/`、`character/`、构建脚本等**共享改进**上有选择地 cherry-pick。

---

## 五、分支管理约定

- `main`:全功能单端(含日报改造)。长期主线。
- `feat/client-server-split`:端云分离。周期性 `merge main`。
- `feat/live2d-only`:纯桌宠。仅按需 cherry-pick `pet/` 相关提交。
- 提交信息:`feat/fix/refactor/chore: 中文描述`。
- 每个分支的差异点集中在少数模块,避免大范围冲突。

---

## 六、落地顺序

1. [main] 日报改造:会话聚合 + 预算 + 事件去重(本文件同批提交骨架)。
2. [feat/client-server-split] 拆 `xiaohan-agent` bin + ingest 路由 + 本地上报客户端。
3. [feat/live2d-only] 剥离 analysis/report/ai/wechat,精简前端与 Cargo。
