# 02 · 是否需要截图分析：实现方式、成本与替代方案

**日期**：2026-07-02  
**状态**：讨论中（未进计划/代码）

## 问题

讨论是否需要截图分析功能；若做，静默截图后送 AI 分析的实现路径；时间成本与 API 费用；是否有更好的用户行为分析方式。

## 结论摘要（讨论稿）

| 维度 | 判断 |
|------|------|
| MVP 是否必需 | **否** — 已有 window_title + app 已能支撑 60–70% 日报价值 |
| Phase 2 价值 | **高** — 解决「同一应用内做什么」的语义缺口 |
| 推荐路径 | **分层**：元数据优先 → 按需截图 → 本地/规则兜底 |
| 成本可控关键 | 降频、差异触发、降分辨率、批处理、仅 ambiguous 窗口 |

---

## 1. 截图分析解决什么问题？

**已有能力**（小寒日报当前）：每 2s 采样前台 `app_name` / `exe_path` / `window_title`，空闲不计入，SQLite 持久化。

**缺口**：同一应用内活动语义不可见。例如 Edge 里写文档 vs 刷 B 站，VS Code 改 A 项目 vs B 项目，Figma 做稿 vs 看稿——窗口标题有时够用（「xxx - Google 文档」），经常不够（「Microsoft Edge」「Visual Studio Code」）。

截图 + 视觉 AI 的价值：**把像素变成可检索、可汇总的文字描述**，从而生成「今天完成了什么」而非「今天用了哪些 exe」。

---

## 2. 典型实现路径（参考小黑日报类产品的逻辑）

```
定时/事件触发 → 截屏(多显示器可选) → 本地预处理 → 视觉模型 API → 结构化描述 → 删原图 → 入库
```

### 2.1 触发策略

| 策略 | 说明 |
|------|------|
| 固定间隔 | 如每 3–5 分钟一张；实现简单，成本可预测 |
| 应用切换 | 新前台窗口稳定 N 秒后截一次；与现有 poller 自然结合 |
| 标题变化 | 同 app 但 title 变（浏览器切 tab）再截 |
| 空闲暂停 | 已有 90s idle，截图 pipeline 同步暂停 |
| 排除列表 | 密码管理器、银行、私密应用不截 |

### 2.2 预处理（降本 + 隐私）

- 缩放到 720p 或更低；JPEG/WebP 质量 60–75
- 可选：只截活动显示器；多屏合并为一张长图或各截一张
- 可选：模糊任务栏/通知区；检测全黑/锁屏/屏保则跳过
- **分析完即删**（小黑日报官网承诺；本地只留 text + category）

### 2.3 AI 调用

- 输入：单张图 + 短 system prompt（输出 JSON：`category`, `summary`, `keywords`）
- 模型：GPT-4o-mini / Claude Haiku 带 vision，或国产视觉模型
- 异步队列：独立 worker 线程，不阻塞采集；失败重试 + 丢弃策略

### 2.4 与现有架构衔接

- 新表 `activity_insights`：`segment_id` 或时间戳关联、`summary`、`category`、`source=screenshot|metadata`
- 日报生成：segments 聚合 + insights 文本 → 一次 cheap LLM 调用（纯文本）出 Markdown

## 3. 时间成本

| 环节 | 典型耗时 | 用户感知 |
|------|----------|----------|
| Win32 截屏 | 50–200 ms | 无（后台） |
| 压缩编码 | 50–100 ms | 无 |
| API 往返 | 1–8 s/张（模型/网络） | **无**（异步） |
| 写库 | <10 ms | 无 |

**瓶颈在 API 延迟，不在本地**。队列积压时：丢弃最旧 / 合并同 app 连续截图 / 降频即可。

**CPU/内存**：Tauri + Rust 下截图线程 + 小队列，通常可控制在可接受范围；需实测 WebView2 同进程时的 GDI 截屏开销。

---

## 4. API 费用粗算

假设工作日有效采集 **8 小时**，截图策略 **每 5 分钟 1 张**：

- 张数：8 × 60 / 5 = **96 张/天**
- 单张 token（缩略后）：约 500–1500 input tokens（视模型计费方式；有的按像素档）

**GPT-4o-mini 量级（2025–2026 参考）**：

- 约 $0.00015–0.001 / 张 → **$0.01–0.10 / 天**
- 月约 **$0.3–3**（个人可接受）

**更高质量模型（GPT-4o / Sonnet vision）**：

- 约 $0.005–0.02 / 张 → **$0.5–2 / 天** → 月 **$10–40**

**更激进间隔（每 10 分钟）**：费用减半；**仅 app 切换时**：可能 **20–40 张/天**，费用再降 50–70%。

**文本日报生成**（纯文本 LLM，不含 vision）：整份日报一次约 $0.01–0.05，相对截图可忽略。

> 费用结论：**可控，但取决于间隔与模型**；产品应让用户自选模型/间隔，并在设置页展示「预估日成本」。

## 5. 有没有更好的方式？

### 5.1 优先增强「元数据层」（低成本，建议先做）

| 手段 | 成本 | 收益 |
|------|------|------|
| **窗口标题解析** | 零 | 浏览器、Office、IDE 标题常含文件名/站点 |
| **浏览器 URL**（扩展或 UI Automation） | 低开发 | 精确到页面级 |
| **规则分类** | 零 | exe/title 关键词 → 开发/会议/文档/娱乐 |
| **纯文本 LLM** | 极低 | 把当日 segments JSON 送 cheap model 写日报 |

对很多打工人，**「应用 + 标题 + 时长」+ 一次文本 LLM** 已能生成 80% 可用日报，无需 vision。

### 5.2 截图的替代/折中

| 方案 | 说明 |
|------|------|
| **仅 OCR 标题栏/活动区** | 比全屏 vision 便宜，但实现复杂、易漏 |
| **本地小模型**（LLaVA、Moondream 等） | 无 API 费，占 GPU/CPU，质量参差 |
| **按需截图** | 仅当 title 为空/泛化（「Edge」「设置」）时触发 |
| **用户手动锚点** | 一键标记「开始写方案」；零 AI 费，体验差 |
| **工具集成** | Git commit、Calendar、Slack——精准但覆盖面窄 |

### 5.3 推荐分层策略（产品向）

```
L0  元数据聚合（现有）           → 时间线、排行、热力图
L1  规则 + 标题解析              → 自动 category
L2  文本 LLM 日报                → 默认开启，用户自备 API Key（密码本已有）
L3  按需 / 低频截图 vision       → 可选，设置里开关 + 间隔 + 排除列表
L4  浏览器扩展 URL               → Phase 2.5，性价比极高
```

**不建议 MVP 默认全开静默截图**：隐私敏感、合规与用户信任成本高；应 **opt-in + 明确告知 + 即时销毁**。

## 6. 隐私与信任（与官网对齐）

[小黑日报](https://www.xiaoheiribao.com/?s=bilibili2#features) 卖点：截图分析后即删、数据本地、直连 AI。

若小寒日报做截图，建议同等或更严：

- 默认关闭；开启前二次确认
- 本地队列加密（可选）；上传 TLS 直连 provider
- 分析完成删除 bitmap；DB 只存 summary
- 排除列表 + 全局快捷键暂停
- 设置页显示「今日已分析 N 张、预估费用」

---

## 7. 决策建议（供后续进计划）

| 选项 | 适合场景 |
|------|----------|
| **A. 不做截图，只做 L1+L2** | 快速上线 AI 日报，成本最低 |
| **B. 按需截图（L3 保守）** | 平衡语义与成本，对齐竞品差异化 |
| **C. 高频截图（竞品默认）** | 日报质量最高，需强隐私叙事 + 费用透明 |

**建议顺序**：L2 文本日报 → L4 浏览器 URL → L3 按需截图。每步可独立验收，避免一次把 vision pipeline 做重。
