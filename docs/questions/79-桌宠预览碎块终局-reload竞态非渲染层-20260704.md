# 79 — 桌宠预览碎块终局：reload 竞态（非渲染层）

**日期**: 2026-07-04
**主题**: doc 78 单轨重构后预览仍碎块；根因在入口时序，最终修复

## 现象

doc 76/77/78 围绕「设置页动作列表预览碎块」排查三轮后，doc 78 把 `spinePet.ts` 重构为单轨模型并删除整套骨骼守卫。单轨后普通交互（点击 `touch`、随机 `dance`/`sleep`）正常，但**设置页逐一预览动作时仍出现图片碎块**，无法组装出完整模型。doc 78 自述「待人工验证」，本次验证未通过。

## 三次误区（doc 76 / 77 / 78）

三轮排查都把根因锁定在 **Spine 渲染层**，用越来越精细的手段在骨骼/附件/deform 内部打转：

- **doc 76**：判断「柴郡 setup 本身就是散开的」+ 误删 `assembledBones` 字段 → 引入双轨（track 0 循环 idle、track 1 播动作）+ idle 种子 + 姿态守卫。
- **doc 77**：发现双轨 idle 叠加会让首帧后约 51/68 根骨骼偏离 → 删 idle 种子，改为 track 1 直接播动作 + 每帧强制把 68 根骨骼拉回 idle 快照。
- **doc 78**：用真实骨骼数据做头无仿真，证伪「setup 散开」，并指出守卫锁死骨骼却放任 deform/IK 继续生效才是元凶 → 删整套守卫，回归单轨 `playOneShot`。

方向很专业，但落点错了：单轨消除了守卫导致的 deform 飞散，预览却依旧碎块，证明渲染层不是全部原因。

## 正确推导思路

抓住最关键的不变量——**普通动作 OK、预览不 OK**——沿调用链追**入口时序**，而非渲染内部：

1. 普通点击：`stage mousedown` → `handleClick` → `playTap` → `playOneShot`，spine 处于稳定运行态。
2. 预览：`PetActionSettings.previewAnimation` 先 `await xiaohan.petShow()`，再 `await xiaohan.petPreviewAnimation(name, loop)`。
3. 两条路径**最终都走同一个 `playOneShot`**，差异只能在入口状态/时序。
4. 追 `petShow` → Rust `show_pet` → 函数末尾无条件 `nudge_pet(app)` 发出 `pet-reload` 事件。
5. 前端 `reloadPet()` 收到事件后 `pet?.dispose()` + `new SpinePet()` 整体重建。
6. 紧随其后的 `pet-preview-animation` 事件在 reload 进行中到达，此时 `pet` 仍指向**即将被销毁的旧实例**。
7. `runPreviewAnimation` → `previewPlay` → `playOneShot` 作用在正在被 dispose 的 spine 上，那一帧 `setAnimation` + `update(0)` + `render` 渲染出骨骼/attachment 飞散的碎块，随后旧实例被销毁。

关键洞察：预览比普通点击**多了一步 `petShow()`**，而这一步会触发 SpinePet 整体重建。碎块不是渲染逻辑的固有缺陷，而是预览事件打在了正在销毁的实例上。

## 真正根因

`show_pet` 末尾的 `nudge_pet(app)`（发 `pet-reload`）对「仅为预览而显示」的场景是有害的：它与紧随其后的 `pet-preview-animation` 事件竞态，`previewPlay` 作用于即将销毁的旧 SpinePet 实例。普通点击不经 `show_pet`，故不受影响——这与症状分布完全吻合。

## 修复（`src/components/PetActionSettings.tsx`）

在 `previewAnimation` 中移除 `await xiaohan.petShow()`：

```ts
const previewAnimation = useCallback(
  async (name: string) => {
    try {
      // 不调 petShow：show_pet 末尾会发 pet-reload 触发 SpinePet 整体重建，
      // 与紧随其后的预览事件竞态，previewPlay 会作用在即将销毁的旧实例上导致渲染碎块。
      // petPreviewAnimation 内部已 ensure+show 窗口，路径与普通点击一致。
      const loopPreview =
        isLikelyIdleName(name) &&
        (name === layout.idleAnimation || name === layout.returnIdleAnimation);
      await xiaohan.petPreviewAnimation(name, loopPreview);
    } catch (e) {
      setFeedback(parseApiError(e, "预览动作"));
    }
  },
  [layout.idleAnimation, layout.returnIdleAnimation, setFeedback],
);
```

依据：`petPreviewAnimation` 对应的 Rust 命令 `preview_animation` 内部已做 `ensure_pet_window` + `win.show()` + `set_focus`，且 `ensure_pet_window` 在窗口已存在时直接 `return Ok(())`、不重建、不发 reload。因此去掉前端 `petShow()` 后，预览路径与普通点击完全一致，不再触发 reload 竞态。

边界已覆盖：桌宠被 `pet_hide` 隐藏（窗口 hide 不销毁）时，`preview_animation` 的 `win.show()` 会恢复显示，spine 未销毁，`previewPlay` 直接作用；窗口被 destroy 时，`ensure_pet_window` 重建窗口，`on_page_load` 发 reload，前端 `reloadPet` 重建后由 `pendingPreview` 机制在 `initSpine` 末尾消费预览——该路径设计上已处理。

## 验证

- `tsc --noEmit`：零错误。
- 启动 `tauri:dev`，设置页逐一预览带 deform 的动作（`attack`/`motou`/`victory`/`stand` 等）→ **不再碎块**，用户确认成功。

## 反思

- 76/77/78 三轮都假设根因在 Spine 渲染层，用越来越精细的骨骼仿真证伪前一轮，方向专业但落点错了。
- 真正的判据从一开始就在现象里：「普通动作 OK vs 预览不 OK」。两者最终都经过同一个 `playOneShot`，分歧点只能在**入口时序与共享状态的竞态**，不在核心函数内部。
- 教训：当两条路径共享同一核心函数、一条正常一条异常时，先查入口时序与实例生命周期（是否有销毁/重建/暂停竞态），再查核心函数内部。渲染层再复杂，若两条路径都经过它，它通常不是分歧点。
- 副产物：doc 78 的单轨重构本身是对的（守卫锁骨骼放任 deform 确实有害），保留；本次只在入口去掉多余的 `petShow()`，两者叠加后预览与普通交互都正常。
