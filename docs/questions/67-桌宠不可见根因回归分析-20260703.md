# 桌宠不可见根因回归分析

## 首次提问

**docs/questions/61**（2026-07-03）：用户第一次报告「桌宠依旧看不到」。

当时认定的根因：
1. 全屏误判（最大化窗口被 `sync_fullscreen_visibility` 立刻 hide）
2. dev 模式 Vite 未就绪就创建窗口 → 空白透明
3. 本进程前台不应触发全屏隐藏

当时操作：`win32.rs` 加 `IsZoomed` / `is_foreground_own_process`；`sync_on_startup` 异步重试 `show_pet`。

## 当前 Bug 所在

**docs/questions/65 与 docs/questions/62 互相矛盾，65 撤销了 62 的核心修复。**

Tauri 2 ACL 按 **origin** 匹配 capability，不仅看 window label：

- `pet-capability`：`windows: ["pet"]`，`local: true`（默认），**无 `remote.urls`**
- `WebviewUrl::External("http://127.0.0.1:1420/pet.html")` → origin 为 **remote**
- 桌宠页 `invoke("pet_get_config")` 等 → **`pet_get_config not allowed on origin [remote: ...]`**
- Spine/静态图无法加载 → 窗口透明 → **肉眼不可见**

`allow-app-commands` 已挂在 pet capability 上，但 **origin 不匹配时仍拒绝**，与 62 文档描述一致。

65 中 `prepare_pet_webview` 的 `win.navigate(dev_url)` 会再次把 webview 切到 remote origin，加剧问题。

## 验证

1. 完全退出后 `scripts\start-dev.bat`
2. 约 2–5s 右下角：粉色「桌宠加载中…」→ 柴郡

## 维护规则

**禁止**在 dev 下对 pet 窗口使用 `WebviewUrl::External(localhost)`，除非同时在 `pet.json` 增加匹配的 `remote.urls`（不推荐，扩大攻击面）。
